# Подписать

Как правильно подписывать элементы интерфейса, чтобы VoiceOver читал их понятно и корректно.

## Сделать видимым

> Note: Текстовая надпись – это базовый элемент доступности.

![Справочник жестов VoiceOver.](labeling-text)

Обычные подписи уже адаптированы для VoiceOver: их размер задаст фрейм фокусу, а текст в лейбле — это текст самой надписи.

Если вы рисуете текст через CATextLayer, то информация о доступности теряется. Восстановить её можно самостоятельно, указав параметры. Разберем такой случай и посмотрим, что UILabel делает за нас.

Для начала, отметим, что элемент доступен, а значит на него можно поставить фокус. Если значение будет false, то VoiceOver попытается найти доступный элемент среди дочерних.

```swift
isAccessibilityElement = true
```

Затем подписать элемент, для описания возьмем текст в элементе. Этот текст VoiceOver прочитает, когда фокус попадет на контрол.

```swift 
accessibilityLabel = text
```

В конце нужно задать фрейм элемента: так фокус будет виден на экране и можно будет навести касанием. Фокус нужен не только для незрячих и VoiceOver, но и для Switch Control.

```swift
accessibilityFrame = frameInScreenCoordinates
```

Готово, доступность в самом простом виде заработала.

> Tip: для заголовков еще надо добавлять трейт .header.

## Подписать

VoiceOver читает текст из `accessibilityLabel` ровно так, как он написан. Это значит, что сокращения и символы будут прочитаны буквально, а не так, как вы ожидаете.

![Пицца с названием](labeling-text-advanced)

Например, надпись «30 см» будет прочитана как «тридцать сэмэ», а не «тридцать сантиметров». Цена «30 ₽» превратится в «тридцать знак рубля» вместо «тридцать рублей». VoiceOver не умеет раскрывать сокращения и подставлять правильные названия валют — он читает то, что видит.

Чтобы исправить это, задайте `accessibilityLabel` с полным текстом:

```swift
priceLabel.text = "30 ₽"
priceLabel.accessibilityLabel = "30 рублей"
```

Если нужно объединить несколько строк в одну подпись, используйте запятую — VoiceOver сделает небольшую паузу и правильно расставит интонацию:

```swift
cell.accessibilityLabel = "Пепперони, 25 см, тонкое тесто, 625 рублей"
```

Для удобства можно завести структуру, которая хранит видимый и доступный текст рядом:

```swift
struct AccessibleText {
    let visibleText: String
    let accessibleText: String
}

let size = AccessibleText(
    visibleText: "30 см",
    accessibleText: "30 сантиметров"
)

let price = AccessibleText(
    visibleText: "30 ₽",
    accessibleText: "30 рублей"
)
```

Так бэкенд или дизайн-система всегда могут предоставить правильный текст для VoiceOver вместе с отображаемым. Подобное пригодится и для backend-driven-ui.

### Множественное число

В русском языке число влияет на форму слова: 1 рубль, 2 рубля, 5 рублей. Если вы подставляете число в строку, нужно учитывать склонение.

Используйте `Localizable.stringsdict` для правильного склонения:

![Пример файла локализации](localizable)

```swift
String(format: NSLocalizedString("%d рублей", comment: ""), price)
```

Теперь «1» превратится в «1 рубль», «2» — в «2 рубля», а «5» — в «5 рублей».

Кроме того, `NSStringVariableWidthRuleType` помогает выбирать разные тексты в зависимости от доступной ширины. Это полезно, когда в интерфейсе нужна короткая форма, а для VoiceOver — полная:

![Пример описания для разной ширины](width-rule)

## Кнопки

![Убрать ингредиенты, кнопка](labeling-button)

> Note: VoiceOver прочитает «Убрать ингредиенты, кнопка».

Кнопка с текстом читается VoiceOver как **подпись + «кнопка»**. Слово «кнопка» добавляется автоматически, если у элемента стоит правильный трейт. Описание типа элемента находится в конце текста, чтобы подсказать, что можно сделать с элементом.

```swift
button.accessibilityTraits = .button
```

> Warning: Добавлять текст «кнопка» самостоятельно не нужно, за вас это сделает iOS, ей достаточно трейта.


> Note: У стандартного `UIButton` трейт `.button` стоит по умолчанию. Если вы делаете кнопку из `UIView` или `UILabel`, не забудьте добавить трейт вручную.

### Активация кнопки

Нажатие на кнопку в VoiceOver — это двойной тап. Он вызывает метод `accessibilityActivate()`:

```swift
override func accessibilityActivate() -> Bool {
    openDetails()
    return true
}
```

Кнопка вызывает обычное нажатие на себя и возвращает true, если действие обработалось. Если вернется false, то VoiceOver попытается вызывать метод у следующего объекта в иерархии UIView. 

AccessibilityActivate() можно вызывать у любого доступного объекта, лишь бы у него был выставлен isAccessibilityElement = true.

> Note: `accessibilityActivate` обычно прописывать не нужно, VoiceOver вызовет то же действие, что и кнопка. 

### Проверка описания

Обязательно проверяйте, как VoiceOver читает текст кнопки. Сокращения и символы могут озвучиваться не так, как вы ожидаете:

- «См все» прочитается как «эс-эм все» вместо «Смотреть все»
- «179 ₽» прочитается как «сто семьдесят девять знак рубля» вместо «сто семьдесят девять рублей»

Исправляйте подписи:

```swift
button.setTitle("См все", for: .normal)
button.accessibilityLabel = "Смотреть все"
```

![Пример кнопки «купить» из апстора](labeling-button-check)

### Кнопки без текста

![Кнопки-иконки рядом с пиццей](labeling-button-without-text)

Кнопки с иконками — самая частая проблема доступности. Если у кнопки нет текста, VoiceOver попытается прочитать название файла изображения. Результат получается неожиданный:

- Кнопка закрытия с иконкой `ic_close` прочитается как «айсиклоуз, кнопка»
- Кнопка корзины с иконкой `ic_cart` — «айсикарт, кнопка»

Это совершенно непонятно для пользователя. Исправить легко — задайте `accessibilityLabel`:

```swift
closeButton.accessibilityLabel = "Закрыть"
cartButton.accessibilityLabel = "Корзина"
infoButton.accessibilityLabel = "Пищевая ценность"
```

### Таббар

![Таббар фейсбука](labeling-tabbar)

Отдельная история — кастомные `UITabBarItem`. Если вы используете собственные элементы в таб-баре, они тоже могут остаться без подписей. Без `accessibilityLabel` табы будут читаться как имена файлов иконок или вообще не будут озвучены.

Например, в Facebook таб-бар может прочитаться как набор файлов вместо понятных «Лента», «Друзья», «Профиль». Всегда задавайте подписи для элементов таб-бара:

```swift
let feedTab = UITabBarItem(
    title: "Лента",
    image: UIImage(named: "ic_feed"),
    tag: 0
)
// title автоматически станет accessibilityLabel
```

Если `title` скрыт визуально, задайте `accessibilityLabel` отдельно:

```swift
feedTab.accessibilityLabel = "Лента"
```

## Изображения

![Экран описания пиццы](labeling-images)

По умолчанию картинки скрыты от VoiceOver: у `UIImageView` свойство `isAccessibilityElement` равно `false`. Это правильное решение для большинства случаев — декоративные изображения не несут смысла и только замедляют навигацию.

Если изображение важно для понимания, сделайте его доступным и добавьте трейт `.image`. Например, на экране описания пиццы можно объединить картинку и заголовок, потому что по смыслу это одно и тоже. 

```swift
imageView.isAccessibilityElement = true
imageView.accessibilityTraits = [.image, .header]
imageView.accessibilityLabel = "Пицца Пепперони"
```

iOS умеет распознавать содержимое изображений и может озвучить описание автоматически, но лучше не полагаться на это — задавайте подписи сами.

Декоративные изображения скрывайте явно:

```swift
decorativeImage.isAccessibilityElement = false
```

Бейджи и пометки на картинках — особый случай. Если на карточке пиццы есть бейдж «NEW», не делайте его отдельным доступным элементом. Лучше добавьте информацию в подпись основного элемента:

```swift
// Плохо: бейдж как отдельный элемент
badgeLabel.isAccessibilityElement = true
badgeLabel.accessibilityLabel = "NEW"

// Хорошо: информация в подписи карточки
cell.accessibilityLabel = "Новинка, Пепперони, 625 рублей"
```

Так пользователь сразу услышит всю информацию о пицце, а не будет переключаться между элементами.

## Как называть элементы?

Хорошая подпись обладает несколькими свойствами:

1. **Краткое описание.** Одно слово или короткая фраза: «Добавить», «Корзина», «Настройки». Незрячие слушают VoiceOver на высокой скорости — длинные описания замедляют работу.

2. **Без указания типа.** Не пишите «кнопка Закрыть» или «картинка Пепперони» — тип элемента добавляется автоматически через трейты. Если написать «Кнопка Закрыть», VoiceOver прочитает «Кнопка Закрыть, кнопка».

3. **Начинается с заглавной буквы.** VoiceOver может использовать регистр для интонации.

4. **Без точки в конце.** Точка создаёт ненужную паузу. Если нужно перечислить несколько свойств, используйте запятую.

5. **На языке пользователя.** Если интерфейс на русском — подписи на русском. «NEW» в `accessibilityLabel` прозвучит как «нью», лучше написать «Новинка».

6. **Составные подписи через запятую.** VoiceOver делает короткую паузу на запятой — используйте это для структурирования:

```swift
cell.accessibilityLabel = "Пепперони, 25 см, тонкое тесто"
```

> Tip: Хорошая проверка: представьте, что вы описываете элемент по телефону другу. Как бы вы назвали его одним-двумя словами?

## Практика

Включите VoiceOver и пройдитесь по вашему приложению. Проверьте следующее:

1. **Найдите неподписанные элементы.** Послушайте каждый элемент на экране — если VoiceOver молчит или читает что-то непонятное вроде имени файла, элементу нужна подпись.

2. **Проверьте кнопки.** Убедитесь, что все кнопки имеют трейт `.button` — VoiceOver должен произносить слово «кнопка» после подписи. Если нет — добавьте трейт.

3. **Найдите проблемы с сокращениями.** Послушайте, как читаются сокращённые слова и символы валют. «См», «шт», «₽», «$» — все они могут озвучиваться неожиданно.

4. **Проверьте изображения.** Декоративные картинки не должны попадать в фокус VoiceOver. Важные изображения должны иметь осмысленные подписи.

5. **Послушайте составные элементы.** Ячейки с несколькими надписями — озвучиваются ли все части? Не теряется ли информация?
